# C++ / Qt 审查方案

## 审查重点

优先关注以下问题：

- 原始指针、父子对象关系、手动 `delete`、悬空引用、重复释放等生命周期问题
- 信号槽连接方式错误、重复连接、跨线程 UI 访问、连接类型不当
- 共享状态线程安全、锁粒度、死锁风险、worker 对象设计问题
- 容器失效、迭代器误用、隐式共享副作用、热路径不必要拷贝
- 返回值未检查、错误传播不足、文件和网络处理脆弱
- 边界检查、整数转换、编码处理、未定义行为
- QFile、socket、timer、model、动态创建控件等资源泄漏
- 在 GUI 主线程执行重计算或阻塞 IO 导致界面卡死
- 解析、线程、生命周期敏感代码缺少测试

## 必查项

1. **对象生命周期**
   - 检查 QObject 派生对象是否有明确 owner
   - 检查 lambda、回调、异步任务中是否捕获了可能失效的引用或指针
   - 检查是否存在手动释放和父对象自动释放并存的重复管理

2. **信号槽与线程**
   - 检查是否从非 GUI 线程访问界面对象
   - 检查跨线程连接是否应使用 queued connection
   - 检查是否重复 connect 导致槽函数被多次触发

3. **资源管理**
   - 检查文件、网络连接、定时器、模型对象是否成对创建与释放
   - 检查异常路径、提前返回路径上是否有资源泄漏
   - 优先建议使用 RAII 替代手工清理

4. **边界与健壮性**
   - 检查数组、容器、字符串索引是否越界
   - 检查整数窄化转换、符号位转换、长度计算是否安全
   - 检查外部输入、文件内容、协议数据是否经过合法性判断

5. **性能与 UI 响应**
   - 检查主线程中是否有明显阻塞操作
   - 检查频繁信号触发下是否存在高成本拷贝或重复刷新
   - 检查模型/视图更新是否过于粗暴导致界面抖动或卡顿

## 常见高风险信号

- `new` 出来的对象没有 parent，也没有清理路径
- `QObject::connect` 重复执行却没有防重
- 在线程回调中直接更新 QWidget
- 信号槽 lambda 捕获局部引用并在异步路径中使用
- 文件/网络错误只打印日志，不向上反馈失败
- 容器迭代过程中删除元素却继续使用旧迭代器

## 输出建议

指出 Qt / C++ 问题时，优先说清楚：
- 生命周期在哪个路径上失控
- 线程归属或连接方式为什么危险
- 是否会造成崩溃、卡死、泄漏或偶发错误
- 推荐的 Qt 风格修复方式，例如 parent ownership、RAII、queued connection、moveToThread 等
