# Vue 审查方案

## 审查重点

优先关注以下问题：

- 响应式状态使用错误，例如 props 直接修改、ref/reactive 混用导致状态失真
- watch、computed、生命周期钩子使用不当
- 异步请求竞态、重复提交、加载态和错误态缺失、Promise 未处理
- 表单校验、条件渲染、列表 key、路由守卫是否存在逻辑问题
- `v-html`、本地存储、URL 拼接、token 处理是否有安全风险
- 定时器、事件监听、订阅、第三方实例是否在卸载时清理
- UI 状态与服务端状态是否混杂，导致数据不一致
- 渲染路径中的高成本计算、频繁重渲染
- 前后端接口契约是否对齐
- 复杂组件或 composable 是否缺少可测试性

## 必查项

1. **状态管理**
   - 检查 props 是否被直接修改
   - 检查 `ref`、`reactive`、`computed` 是否被正确访问和更新
   - 检查多个状态源是否可能产生不一致

2. **异步与交互**
   - 检查接口请求是否处理 loading、error、finally
   - 检查按钮是否有防重复提交控制
   - 检查快速切换路由、快速点击时是否可能发生竞态覆盖

3. **生命周期清理**
   - 检查 `onMounted` 中创建的监听器、定时器、第三方实例是否在 `onUnmounted` 中释放
   - 检查组件销毁后是否仍可能更新状态

4. **安全与数据展示**
   - 检查是否使用不安全的 `v-html`
   - 检查是否把 token、敏感字段写入不安全存储或拼接进 URL
   - 检查用户输入是否直接用于危险渲染

5. **列表和表单**
   - 检查 `v-for` 的 key 是否稳定唯一
   - 检查表单校验、禁用态、提交反馈、失败回滚是否完整
   - 检查分页、筛选、搜索条件是否会导致状态错乱

## 常见高风险信号

- 请求函数里没有错误处理，失败后界面仍显示成功态
- `watch` 内部再次修改其监听源导致循环更新
- 路由切换后旧请求回写新页面数据
- `v-for` 使用索引作为 key 导致复用错误
- 组件卸载后定时器仍在运行
- 直接渲染后端返回的 HTML

## 输出建议

当指出前端问题时，尽量明确：
- 触发条件，例如快速点击、切页、接口失败、组件销毁后返回结果
- 页面表现，例如重复提交、旧数据覆盖、新旧状态不一致
- 建议修复方式，例如增加取消请求、唯一 key、卸载清理、防抖或提交锁
