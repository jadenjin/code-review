# Java / Spring Boot 审查方案

## 审查重点

优先关注以下问题：

- Controller、Service、Repository 分层是否清晰，是否出现职责混乱
- 请求参数、DTO、外部输入是否做了必要校验
- 事务边界是否正确，例如部分写入、遗漏 `@Transactional`、事务范围过大
- 安全问题：权限校验缺失、越权访问、SQL 注入风险、路径处理不安全、日志泄露敏感信息
- 空值处理、Optional 误用、集合访问越界、日期和时区错误
- 异常处理是否吞错、误导调用方、返回不正确状态码
- ORM 使用是否可能引发 N+1 查询、序列化懒加载、分页错误、低效批处理
- 共享状态、异步任务、定时任务是否存在并发不安全或上下文丢失
- 配置、Bean 生命周期、依赖注入是否存在明显错误
- 高风险业务逻辑是否缺少测试覆盖

## 必查项

1. **输入校验**
   - 检查 `@Valid`、`@Validated`、参数约束注解是否存在且真正生效
   - 检查空字符串、null、非法枚举值、边界数值是否有处理

2. **权限与安全**
   - 检查接口是否遗漏鉴权或角色限制
   - 检查是否直接信任前端传入的用户 ID、状态值、资源路径
   - 检查是否在日志、异常中输出敏感数据

3. **事务与一致性**
   - 检查多个写操作是否应纳入同一事务
   - 检查异常场景下是否可能出现部分成功、部分失败
   - 检查异步调用是否破坏事务一致性

4. **异常与返回**
   - 检查异常是否被吞掉，只打印日志不向上抛出
   - 检查 Controller 返回码是否与实际结果一致
   - 检查是否把内部异常包装成误导性成功结果

5. **持久层与性能**
   - 检查循环中查询数据库、逐条更新、未分页读取全量数据
   - 检查实体直接序列化是否触发懒加载问题
   - 检查复杂列表接口是否存在明显性能隐患

## 常见高风险信号

- Service 中同时混杂鉴权、参数组装、数据库写入、HTTP 返回逻辑
- Controller 中直接写复杂业务逻辑
- Repository 调用出现在深层循环中
- 使用共享可变对象保存请求态数据
- 使用 `catch (Exception e)` 后仅打印日志继续返回成功
- 使用前端传入字段直接控制权限、状态流转或数据库查询条件

## 输出建议

当发现问题时，优先描述：
- 具体出错场景
- 可能造成的业务影响
- 更稳妥的修复方式

避免只说“建议优化”而不说明风险。
