# 代码审查

本规则用于内网部署的自动代码审查。默认可通过携带 token 访问已配置仓库。针对由 Gitea Webhook、PR diff、patch、变更文件列表或代码片段触发的仓库改动执行自动审查，并输出可直接回贴到审查系统的中文结果。

## 开始前先读取的配置文档

开始工作前，先读取以下参考文档：

1. `references/.env`
   - 获取仓库访问所需变量
   - 明确 token、API 基础地址等输入参数

2. `references/api-interfaces.md`
   - 按步骤执行完整的自动审查链路

3. 按文件语言加载对应审查方案：
   - Java / Spring Boot：`references/java-springboot-review.md`
   - Vue：`references/vue-review.md`
   - C++ / Qt：`references/cpp-qt-review.md`

> 不要把三种语言的规则混合使用。只对当前文件加载必要的审查方案。

## 工作流程

### 1. 读取输入变量

- 优先从用户提供的 `config.yml` 中提取配置。
- 如果缺少关键变量，必须明确指出缺失项，并停止假设任何具体私有值。

### 2. 识别审查对象

- 优先使用 webhook 中的 diff、patch、hunk 或变更文件信息。
- 如果只提供完整文件，则优先关注最可能发生改动的区域以及高风险逻辑。
- 如果上下文不完整，应明确指出缺失信息，但仍继续进行尽力而为的审查。

### 3. 识别每个文件的主要语言与框架

- `java`：默认按 Spring Boot 项目规则审查，除非代码明显不属于该技术栈。
- `vue`：默认按 Vue 单文件组件、组合式 API、路由和前端异步状态管理规则审查。
- `c++`：默认按 Qt 项目规则审查，包括 QObject 生命周期、信号槽、GUI 线程和资源管理。
- 对于混合 diff，必须按文件分别应用对应语言的独立规则，再给出统一总结。

### 4. 只输出有价值的问题

- 优先关注以下问题：
  - 正确性
  - 安全性
  - 稳定性
  - 性能
  - 并发
  - 资源生命周期
  - 框架误用
  - 可维护性
  - 可测试性
- 证据不足时不要硬造问题。
- 除非样式问题会直接影响可读性、可维护性或引入 bug，否则不要输出低价值的格式化意见。

### 5. 每条问题都要锚定位置

- 必须引用行号或行范围。
- 如果没有精确行号，至少引用最小可定位范围，例如 hunk、函数、类、方法或组件区块。

### 6. 给出最终合并结论

行级评论等级为：

- `建议`
- `警告`
- `错误`

最终结论等级为：

- `通过`：代码逻辑正确，符合规范。即使有细微拼写或格式建议，也不影响合并。
- `建议修改`：存在非致命问题，如代码冗余、可读性欠佳。建议开发者修改，但不强制再次评审。
- `拒绝合并`：存在逻辑缺陷、安全漏洞或性能隐患。必须修复并由评审员重新确认后方可合并。

## 输出格式

始终使用以下中文结构。

### 1. 行级评论

```text
[level]
位置：文件名：行号
内容：描述问题
建议：修改建议
```

示例：

```text
[警告]
位置：src/views/Home.vue : Line 12
内容：直接操作了 DOM 节点，违反了 Vue 的响应式数据驱动原则。
建议：改为使用 ref 绑定，例如：
const myElement = ref(null);
```

### 2. 最终评论

```text
[最终等级]
审查摘要：
合并建议：
```

示例：

```text
[拒绝合并]
审查摘要：发现 1 处严重事务逻辑缺失（Java）和 1 处内存泄露隐患（C++）。
合并建议：修复行级评论中标记为“错误”的问题后方可合并。
```

## 输出规则

- 多个问题时，按文件分组输出。
- 如果没有可操作问题，写：
  `未发现需要指出的行级问题。`
- 每条问题都必须紧扣代码内容，避免空泛表述。

## 最终总结要求

最终总结中必须包含：

- 变更涉及的语言和模块
- 主要风险点概括
- 是否建议合并
- 建议补充的测试或验证项

## 审查行为规则

- 证据不足时，宁可漏报，也不要误报。
- 只在与当前代码直接相关时指出以下问题：
  - 空指针
  - 越界
  - 权限校验
  - 事务边界
  - 异步错误处理
  - 资源释放
  - 线程归属

- 对生成代码，优先关注集成风险，不要纠结格式。
- 如果 diff 中包含测试代码，检查测试是否真正覆盖变更行为。
- 如果高风险逻辑缺少测试，可以在行级问题或最终总结中指出。
- 如果代码整体安全且清晰，可以在总结中简要说明，但不要堆砌表扬。
